# Investigation: Blender → cd.iff Pipeline

**Date:** 2026-04-13  
**Context:** The original World Foundry content pipeline ran through 3DS Max (1995–2001).
The Blender plugin (`wf_blender`) has been rebuilt for per-object OAD property editing,
but there is no path from a Blender scene to a playable game. This document maps the
existing pipeline and proposes the new path.

---

## 1. What is cd.iff?

`cd.iff` is the master game data archive — a binary IFF container that the game engine
opens at startup (`wfsource/source/game/game.cc:93-94`). Everything the game needs lives
inside it: levels, shell/boot data, assets.

**Structure** — real example from `wftools/iffcomp-go/testdata/cd.iff.txt`
(generated by `cd.pl` from `levels.txt`):

```
{ 'GAME'
	{ 'TOC'
		'SHEL'	.OFFSETOF( ::'GAME'::'SHEL' )	.SIZEOF( ::'GAME'::'SHEL' )
		'L0'	.OFFSETOF( ::'GAME'::'L0' )	.SIZEOF( ::'GAME'::'L0' )
		'L1'	.OFFSETOF( ::'GAME'::'L1' )	.SIZEOF( ::'GAME'::'L1' )
		'L2'	.OFFSETOF( ::'GAME'::'L2' )	.SIZEOF( ::'GAME'::'L2' )
		'L3'	.OFFSETOF( ::'GAME'::'L3' )	.SIZEOF( ::'GAME'::'L3' )
		'L4'	.OFFSETOF( ::'GAME'::'L4' )	.SIZEOF( ::'GAME'::'L4' )
		'L5'	.OFFSETOF( ::'GAME'::'L5' )	.SIZEOF( ::'GAME'::'L5' )
		'L6'	.OFFSETOF( ::'GAME'::'L6' )	.SIZEOF( ::'GAME'::'L6' )
	}
	{ 'ALGN' .ALIGN( 2048 ) }         ← PSX/CD sector alignment

	{ 'SHEL' [ "shell.aib" ] }        ← boot/meta script
	{ 'ALGN' .ALIGN( 2048 ) }

	{ 'L0'
		{ 'ALGN' .ALIGN( 2048 ) }
		[ "cubemenu.iff" ]             ← level 0
	}
	{ 'ALGN' .ALIGN( 2048 ) }

	{ 'L1'
		{ 'ALGN' .ALIGN( 2048 ) }
		[ "3dtext.iff" ]               ← level 1
	}
	{ 'ALGN' .ALIGN( 2048 ) }

	{ 'L2' { 'ALGN' .ALIGN( 2048 ) } [ "cyberthug.iff"  ] }
	{ 'ALGN' .ALIGN( 2048 ) }
	{ 'L3' { 'ALGN' .ALIGN( 2048 ) } [ "primitives.iff" ] }
	{ 'ALGN' .ALIGN( 2048 ) }
	{ 'L4' { 'ALGN' .ALIGN( 2048 ) } [ "snowgoons.iff"  ] }
	{ 'ALGN' .ALIGN( 2048 ) }
	{ 'L5' { 'ALGN' .ALIGN( 2048 ) } [ "bubba.iff"      ] }
	{ 'ALGN' .ALIGN( 2048 ) }
	{ 'L6' { 'ALGN' .ALIGN( 2048 ) } [ "whitestar.iff"  ] }
	{ 'ALGN' .ALIGN( 2048 ) }
}
```

`cd.pl` generates this from a plain `levels.txt` (one level name per line):

```
cubemenu
3dtext
cyberthug
primitives
snowgoons
bubba
whitestar
```

`iffcomp` compiles `cd.iff.txt` → `cd.iff` (binary). The `.OFFSETOF` / `.SIZEOF`
directives are back-patched by `iffcomp` after all chunks are laid out.

**Build rule** (`wfsource/levels.src/GNUmakefile:56-60`):

```makefile
$(ASSET_DIR)/cd.iff.txt : levels.txt $(ASSET_DIR)/shell.tcl
    perl cd.pl < levels.txt > $@

$(ASSET_DIR)/cd.iff : $(ASSET_DIR)/cd.iff.txt
    cd $(ASSET_DIR) ; iffcomp -o=cd.iff cd.iff.txt
```

## 2. Tool Glossary

| Tool | Language | What it does |
|------|----------|-------------|
| `iffcomp` | C++ + flex/bison | `.iff.txt` (text DSL) → binary IFF |
| `iffdump` | C++ (Rust port done) | Binary IFF → annotated hex dump or iffcomp source |
| `iff2lvl` | C++ (~8.7k LOC) | Binary level IFF → runtime `.lvl` format |
| `max2lvl` | C++ Win32 DLL | 3DS Max plugin: reads scene in-process → `.lvl` |
| `lvldump` | C++ | `.lvl` → human-readable dump |
| `textile` | C++ | TGA/BMP/SGI → texture atlas + UV maps |
| `prep` | C++ | Macro/template processor — drives `.oas` → C codegen |
| `oas2oad` | Rust (done) | `.oas` schema → binary `.oad` (replaces Win32 wpp+wlink chain) |

## 3. iff2lvl vs max2lvl

Same C++ business logic, different host:

```
┌─────────────────────────────────────────────────────────────────┐
│  max2lvl  (wfmaxplugins/max2lvl/)                               │
│                                                                 │
│  Host: 3DS Max Win32 DLL                                        │
│  Input: Max scene graph — in-process, via Max SDK               │
│  Output: .lvl binary (written directly)                         │
│  History: original plugin form, 1995–1999                       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  iff2lvl  (wftools/iff2lvl/)                                    │
│                                                                 │
│  Host: stand-alone command-line (Linux + Win32)                 │
│  Input: binary IFF .lev.bin (pre-compiled from Max's .lev       │
│         text export by iffcomp)                                 │
│  Output: .lvl binary                                            │
│  History: created 1999 to decouple from Max                     │
│  Status: builds on Linux; README calls it "a mess"              │
└─────────────────────────────────────────────────────────────────┘
```

The two directories share ~95% identical source files. `iff2lvl` was extracted from
`max2lvl` so the pipeline could run on Linux with any 3D app exporting IFF.

**iff2lvl does NOT read textual IFF.** It reads binary IFF via `binistream` +
`IFFChunkIter` (`wftools/iff2lvl/level.cc:733`). The textual `.lev` must be compiled
to `.lev.bin` by `iffcomp` first.

## 4. Existing Pipeline (3DS Max era)

```
┌──────────────┐
│  3DS Max     │
│  scene       │
└──────┬───────┘
       │
       ├─── max2lvl (DLL plugin, in-process) ──────────────────────┐
       │                                                            │
       └─── File > Export ──▶ .lev (text IFF)                      │
                                    │                               │
                                    ▼                               │
                             ┌────────────┐                         │
                             │  iffcomp   │                         │
                             └─────┬──────┘                         │
                                   │                                │
                                   ▼                                ▼
                            .lev.bin (binary IFF) ──▶ iff2lvl ──▶ .lvl
                                                         │
                                                    objects.lc
                                                    (OAD index)
```

### Per-level build (generated by `unixmakelvl.pl`)

```
.lev   ──[iffcomp]──▶  .lev.bin
                            │
                      [iff2lvl + objects.lc]
                            │
                            ▼
                          .lvl   ──[textile]──▶  perm.tga
                            │                    asset lists
                            │
                      [prep iff.prp]
                            │
                            ▼
                       .iff.txt  ──[iffcomp]──▶  levelname.iff
                                                      │
                                             (collected into cd.iff)
```

### cd.iff assembly

```
levels.txt
    │
  [cd.pl]
    │
    ▼
cd.iff.txt  ──[iffcomp]──▶  cd.iff   ← game opens this at boot
```

## 5. Current wf_blender State

**What works today** (`wftools/wf_blender/`):

```
Blender object
    │
    ├── Attach Schema (WF_OT_attach_schema)
    │       loads .oad → seeds wf_* custom properties on object
    │
    ├── Edit properties in panel (all field types: Int, Float, Bool, Enum, Str)
    │
    ├── Validate (WF_OT_validate)
    │       wf_core Rust → range-checks all values
    │
    ├── Export .iff.txt (WF_OT_export_iff_txt)
    │       wf_core → text IFF for this object's OAD data
    │
    └── Export .iff (WF_OT_export_iff)
            wf_core → binary IFF for this object's OAD data
```

**What is missing**: any scene-level export. There is no operator that iterates
all objects in a scene and writes a level file.

## 6. Proposed New Pipeline (Blender → cd.iff)

```
┌──────────────────────────────────────┐
│  Blender scene                       │
│                                      │
│  Object A  [wf_schema: actor.oad]    │
│  Object B  [wf_schema: platform.oad] │
│  Object C  [wf_schema: camera.oad]   │
│  ...                                 │
└──────────────┬───────────────────────┘
               │
               │  NEW: WF_OT_export_level_iff_txt
               │  (wftools/wf_blender/export_level.py)
               │
               ▼
        ┌─────────────┐
        │  level.lev  │   text IFF, FOURCC '\0LVL'
        │  (text IFF) │   one '\0JBO' chunk per object
        └──────┬──────┘
               │
               │  iffcomp  (existing)
               │
               ▼
        ┌──────────────┐
        │ level.lev.bin│   binary IFF
        └──────┬───────┘
               │
               │  iff2lvl + objects.lc  (existing C++, builds on Linux)
               │  OR future: lvlcomp-rs (Rust rewrite)
               │
               ▼
        ┌───────────┐
        │ level.lvl │   binary runtime level
        └──────┬────┘
               │
               │  (texture / asset pipeline — unchanged)
               │
               ▼
        ┌─────────────┐           ┌──────────┐
        │ cd.iff.txt  │──[iffcomp]──▶ cd.iff │  ← game opens this
        └─────────────┘           └──────────┘
```

## 7. What the Level IFF Must Contain

From `wftools/iff2lvl/level.cc:729` (`LoadLevelIFFFile`):

```
{ '\0LVL'                        ← top-level chunk
    { '\0JBO'                    ← one per game object
        [object type index]      ← int16: index into objects.lc OAD table
        [x, y, z]                ← fixed32 position
        [x_scale, y_scale, z_scale]  ← fixed32 scale
        [rotation a, b, c, order]    ← wf_euler (3 × uint16 + uint8)
        [collision box min/max]      ← 6 × fixed32
        [oadFlags]               ← int32 bit field
        [pathIndex]              ← int16 (-1 = no path)
        [OADSize]                ← int16
        [OAD data blob]          ← OADSize bytes of packed field values
    }
    { '\0JBO' ... }
    ...
}
```

The `objects.lc` / `objects.p` file maps integer type indices to OAD schema names.
`iff2lvl` also reads the `.oad` files for each referenced type to interpret the OAD data.

## 8. Implementation Plan

### Step 1: New operator `WF_OT_export_level_iff_txt`

**File:** `wftools/wf_blender/export_level.py`

Iterates all scene objects that have `wf_schema_path` set, and emits:

```python
# pseudocode
with open(filepath, 'w') as f:
    f.write("{ '\\0LVL'\n")
    for obj in bpy.context.scene.objects:
        if "wf_schema_path" not in obj:
            continue
        schema = wf_core.load_schema(obj["wf_schema_path"])
        values = _collect_values(obj, schema)
        oad_blob = wf_core.export_iff(schema, values)  # binary OAD data
        # write '\0JBO' chunk: type_index, transform, collision, oad_blob
        write_obj_chunk(f, obj, schema, oad_blob)
    f.write("}\n")
```

Model on: `wftools/wf_blender/operators.py:266` (`WF_OT_export_iff_txt`)

### Step 2: Wire into build system

The existing `wfsource/levels.src/GNUmakefile` already has the `.lev.bin → .lvl`
rules. A Blender-exported `.lev` file just needs to be placed where the Makefile
expects it.

### Step 3: Verify round-trip

```bash
# 1. Export from Blender → produces level.lev (text IFF)

# 2. Compile to binary IFF
iffcomp -o=level.lev.bin level.lev

# 3. Convert to runtime level
iff2lvl level.lev.bin wfsource/source/oas/objects.lc level.lvl

# 4. Inspect
lvldump level.lvl

# 5. Build a minimal cd.iff.txt by hand (or via cd.pl):
cat > /tmp/levels.txt <<'EOF'
mylevel
EOF
perl wfsource/levels.src/cd.pl < /tmp/levels.txt > /tmp/cd.iff.txt

# cd.iff.txt will look like:
# { 'GAME'
#     { 'TOC'
#         'SHEL'  .offsetof( ::'GAME'::'SHEL' )  .sizeof( ::'GAME'::'SHEL' )
#         'L0'    .offsetof( ::'GAME'::'L0' )     .sizeof( ::'GAME'::'L0' )
#     }
#     { 'ALGN' .align( 2048 ) }
#     { 'SHEL' [ "shell.tcl" ] }
#     { 'ALGN' .align( 2048 ) }
#     { 'L0'
#         { 'ALGN' .align( 2048 ) }
#         [ "mylevel.iff" ]
#     }
#     { 'ALGN' .align( 2048 ) }
# }

# 6. Compile cd.iff
cd /path/to/assets/linux
iffcomp -o=cd.iff /tmp/cd.iff.txt

# 7. Run the game — it looks for cd.iff in the current directory
cd wfsource/source/game && ./game
```

## 9. Open Questions

| Question | Impact |
|----------|--------|
| **Room system**: Rooms are explicit game objects with `OAD=Room` — authored in Blender like any other object, not computed. | Resolved |
| **Mesh/asset pipeline**: Phase 1 uses `MODEL_TYPE_BOX` (0) — engine renders the collision bounding box directly via `RenderActor3DBox`, no mesh file or textile needed (`wfsource/source/game/actor.cc:402`). Phase 2 switches to `MODEL_TYPE_MESH` (1) + mesh `.iff` + textile. | Resolved for phase 1 |
| **Coordinate convention**: Blender is Z-up, WF is Y-up. Export applies: `wf_x = bl_x`, `wf_y = bl_z`, `wf_z = -bl_y`. | Resolved |
| **iff2lvl rewrite**: Use existing C++ binary as-is — it builds and works on Linux. Rewrite deferred. | Resolved |
| **Type index table**: `objects.lc` is generated by `prep objects.lcs` expanding `objects.mac` — one name per line, `NullObject` at index 0. Names keep their `objects.mac` casing and are truncated to 8 characters (`Generator`→`Generato`, `Destroyer`→`Destroy`). The exporter strips `.oad` from the schema filename and does a case-insensitive, 8-char-prefix match against `objects.lc` to get the type index. | Resolved |
